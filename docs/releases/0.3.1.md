# TCRN Workflow 0.3.1

`0.3.1` is the fourth accepted release. It changes no engine behaviour: every
verb, `work-annotate` included, is byte-for-byte the `0.3.0` behaviour. What it
completes is the release machinery itself.

## What went wrong in 0.3.0

The `0.3.0` version sweep covered the package manifests, the framework
constant, the READMEs, and the versioning documents — but missed the P8 release
pins: `P8_VERSION` and `P8_TAG` in `scripts/lib/p8-workflow-rc.mjs` still said
`0.2.0`. The consequence is structural, not cosmetic: `verify:p8` refuses the
mismatch (`P8_PACKAGE_VERSION_MISMATCH`), so the `v0.3.0` tree can never build
its own release artifacts, and the `v0.3.0` GitHub Release therefore carries no
source tar, no provenance, no SBOM, and no checksums. Fixing the pins after the
tag cannot repair `v0.3.0` either — artifacts built from a corrected tree would
describe bytes that are not the tagged tree, a provenance for a build nobody
ran. A patch release is the only honest completion.

## What this release does

- Moves `P8_VERSION` / `P8_TAG` to `0.3.1` / `v0.3.1`, with their pinned test
  expectations.
- Registers `docs/releases/0.3.0.md` and `docs/releases/0.3.1.md` in the proof
  route lists (`routeAdditions` and its test fixture mirror), so the proof
  chain admits the release documents added since `0.2.0`.
- Ships the full P8 artifact set on the `v0.3.1` release: source tar,
  `provenance.json`, `release-manifest.json`, `sbom.cdx.json`,
  `checksums.txt` — the assets `v0.3.0` could not produce.

Downstream consumers that pin a release should pin `v0.3.1`. Chains written
under `0.3.0` replay unchanged — the two engines are behaviourally identical.
