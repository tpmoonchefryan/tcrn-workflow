# TCRN Workflow 0.2.0

`0.2.0` is the second accepted release. It does one thing: it makes gate
identity real. A gate whose outcome class is `owner_intent_required` can no
longer be satisfied by whichever actor happens to run the command — closing one
now requires an out-of-band roster naming which actors may close which classes,
and a named actor that roster permits.

The minor version is honest about a behaviour change: on `0.1.0`, satisfying an
`owner_intent_required` gate needed only anchoring minutes. On `0.2.0` the same
transition without a roster is refused with `WORKSPACE_GATE_IDENTITY_REQUIRED`.
Chains written under `0.1.0` replay unchanged — five satisfied owner gates on
two production chains were replayed byte-identical under this engine before the
cut — but the verb itself now asks for more. The class is the opt-in, chosen
per gate at creation; the other four classes behave exactly as before.

## Gate identity, and its stated boundary

The roster is a pins-track authority: a canonical JSON document read through
the same TOCTOU-hardened file reader every other authority uses, bound by a
digest the caller states at the call site (`--identity-authority` plus
`--identity-authority-digest` on `gate-transition`). A satisfied transition
records what was checked — the acting actor and the roster digest in force — in
the gate's extensions, where replay verifies its shape and its consistency with
the event's own signer, and deliberately never re-reads the file.

Three boundary statements ship in
`docs/architecture/agent-integration-v1.md` §9, and the third is pinned by a
test that would fail the day it stopped being true: this delivers authorization
rather than authentication; replay does not defend against a forged
tail-append; and no non-cryptographic replay check could, because event hashes
are unkeyed. Closing that gap is the external-anchor programme, not this
feature.

## Out-of-band authority reaches the CLI

Six pins-track verbs gained optional digest flags
(`profile-resolve`, `profile-authorize`, `context-route`,
`adapter-rollback-plan`, `claude-adapter-rollback-plan`,
`claude-adapter-uninstall`): the caller states the digest it already holds and
the reader verifies it against the bytes on disk, the shape
`--expected-plan-digest` already used. A wrong digest stops at the digest
comparison; supplying an authority both by injection and by flag fails closed
as ambiguous. The five verbs guarded by brand-constructed host contexts
deliberately gained nothing: a flag-supplied host context is a caller assertion
no argument validation can check, and dressing an assertion as an argument is
the conflation the two tracks exist to avoid.

## Smaller, also measured

The push gate runs in ~134s against ~205s before: the two guard-registry
entries that ran whole suites to observe one failure now name the case that
catches their mutation, observed rather than guessed. The recovery stress test
holds the recoverer to the contract the implementation actually makes (at most
one winner, not exactly one). The source lints judge code instead of prose:
comments and string literals are blanked, length-preserving, before the word
scans run.

## Compatibility

- Event-chain format: unchanged. Chains written by `0.1.0` replay under
  `0.2.0` with identical state; the gate-identity extension entry is optional
  forever.
- New reason codes: `WORKSPACE_GATE_IDENTITY_REQUIRED`,
  `WORKSPACE_GATE_IDENTITY_REFUSED`, and the `GATE_IDENTITY_*` authority
  family.
- `supportedAosReleases` remains `[]`.
- The companion helper accepts this release through its own re-pinned
  bootstrap; until that candidate is published, helper installations continue
  to resolve `v0.1.0`.
