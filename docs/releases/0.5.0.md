# TCRN Workflow 0.5.0

`0.5.0` is the seventh accepted release. It adds the sprint / release-train
mechanism: a way to batch Initiatives into a named delivery train and ship them
together, without entangling the timebox axis with the work-scope tree
(`TCRN-CROSS-INIT-008`, EPIC-016).

## Sprint delivery batches (`TCRN-CROSS-INIT-008`, EPIC-016)

A sprint here is a *delivery batch* — a release train — not a work-tree layer. It
is orthogonal to the Initiative → Epic → Story tree, and the two are kept from
entangling structurally.

- **The `Release` kind is open on the CLI create path.** `Release` has been a
  valid protocol kind since `0.1.0`; the create whitelist now admits it (the same
  operating-surface-catches-up-with-the-protocol move that opened `Incident` in
  `0.3.0`). A sprint is a `Release` record, and the graph validator enforces it
  **parentless** — a top-level batch container, structurally parallel to an
  Initiative — so a sprint can never become a parent or child of a work item.
  `Review` and `Knowledge` stay closed to creation.
- **Members attach with `work-annotate --sprint`.** The new flag writes a
  `required:false` `advisory:sprint` extension whose value is a **qualified
  reference** `{workspaceId, workId}` to the sprint's Release record — carried as
  full protocol ids, the `advisory:decided-by` convention. Because it is
  qualified, a member in one partition can ride a train in another (a sub-project
  Initiative on the platform's release train). It is non-binding: it never changes
  status, never gates `done`, and is not checked for referential existence.
- **The read face.** `work-show` surfaces `advisory.sprint`; `work-list --sprint
  workspace:<id>#work:<id>` returns a train's members in the queried workspace
  (a cross-partition train is read by asking each partition). The CLI accepts the
  ergonomic `#`-delimited spelling and stores the structured object.

The engine and the event protocol are unchanged: `advisory:sprint` rides the
existing `work.annotated` operation, adds no registry row, and a workspace that
never sprints is byte-identical to `0.4.0`. As with every advisory key and
`work.annotated` itself, a chain that *uses* `advisory:sprint` requires this
release to replay (the closed anti-smuggling guard), which is why it ships as a
minor bump that readers of such chains pin.

The same guard now runs on the `work.created` reducer branch as well as
`work.annotated`, so a forged create can no longer smuggle a malformed advisory
value past the envelope-only graph validator — a defense-in-depth fix that
covers `advisory:scope` and `advisory:decided-by` too.

## Verification

`verify:p1` (20 gates) and `verify:p8` are green, `guard-check` proves every
registered red-proof still bites (including the three new sprint guards SPRINT-01
through SPRINT-03), and the mechanism ships with sprint create/parentage,
annotation round-trip/replay/forge-rejection, and CLI filter tests.
