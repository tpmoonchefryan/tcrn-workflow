# TCRN Workflow 0.3.2

`0.3.2` is the fifth accepted release. It bundles the Incident create path with
the knowledge-store curation headroom that lets the store hold a real, queried
library instead of scanning near-full on almost no content.

## The Incident create path (`TCRN-CROSS-INIT-004`)

The protocol has validated eight work kinds since `0.1.0` — Initiative, Epic,
Story, Subtask, Review, Incident, Release, Knowledge — but `work-create` only
ever admitted the four planning kinds, so a defect met in passing could only be
recorded as a `Story`. This release opens `Incident` on the CLI create path,
with and without a parent; `Review`, `Release`, and `Knowledge` remain closed
to direct creation. The graph validator already refused a dangling parent for
an unmapped kind, and a regression now pins that guard.

## Knowledge-store curation headroom (`TCRN-CROSS-INIT-005`, EPIC-007)

Three changes turn the knowledge store from a write-only surface that bricks
near ~30 tiny records into one that holds a curated, queryable library:

- **The aggregate cap stops double-charging the derived index.** The index's
  `records` array re-serializes the metadata already counted, so the cap taxed
  every record roughly twice — a store of ~30 mirror records scanned at 97%
  while its bodies were 3% of the bytes. The cap now bounds only the
  source-of-truth bytes (marker + Σ metadata + Σ body) across the scan and the
  create and promotion projections; the index is still rebuilt and validated
  byte-for-byte, and its read stays bounded. Measured headroom for real curated
  cards rose from ~16 to 30-plus.
- **Retire reclaims the body.** A retired record's body is already unreadable
  through `readKnowledgeBody`, so retiring now deletes it as the final durable
  step. Retire frees bytes rather than only a live-record slot. The scan's
  integrity invariant becomes "every body has a matching metadata, and every
  *live* record has its body"; a retired record may lack one.
- **Search matches summaries.** A curated card's searchable substance lives in
  its summary, not its subject; search now scans the summary too.

Each change ships with a red-proven regression: the aggregate cap still fires
once the source-of-truth bytes exceed it, deleting a live record's body is still
rejected, and a summary-only term is now found.

Downstream consumers that pin a release should pin `v0.3.2`. Chains written
under `0.3.1` replay unchanged — no persisted record shape changed.
