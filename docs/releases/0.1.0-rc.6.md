# TCRN Workflow 0.1.0-rc.6

`0.1.0-rc.6` is an immutable unpublished local Workflow release candidate. It is a
correctness and hardening candidate: no new capability surface, no protocol change,
no schema migration. Every prior candidate remains immutable and is superseded for
current-candidate purposes.

It exists because a nine-agent code-quality review of `0.1.0-rc.5` found defects that
the 20-gate verification chain could not see — including three that could destroy
data or lock a Workspace permanently. Those are fixed here, with the failure
reproduced against the shipped bytes before and after each fix.

## Why you would take this candidate

**Three defects in rc.5 could brick local state.** A knowledge store leaked its
mutation claim on any failure that left the process alive, and every later mutation
then answered `KNOWLEDGE_LOCKED` forever with no verb able to clear it. A crash
during lease recovery left a claim recording a `pid` and an expiry that nothing ever
read, so the Workspace could never be opened again. And `conference-append-position`
silently discarded the position author whenever an attestation actor was supplied.

**Fourteen enum admission points accepted values that merely coerced to a legal
member.** `["specified"]`, an object carrying `toString`, and a boxed `String` all
passed. Numbers, `null`, and plain objects were always rejected — so the hole was
narrower than a coercion audit suggests, and the negative cases now test exactly the
shapes that got through rather than the shapes that never did.

**Six ways out of the offline guard and child-process policy are closed**, including
one the review did not name: `execSync` and `execFileSync` were absent from the
guarded API list entirely, so a synchronous exec was governed by nothing at all.

## What is measurably faster

- `verify:privacy` 56.6s to 2.4s (23.9x), which pulls `verify:p1` from 133s to 81s.
  The privacy scan spawned two git processes per object — roughly five thousand —
  and now reads one `cat-file --batch-all-objects --batch` stream. Scanned-entry and
  object counts are byte-identical to the baseline.
- `validateEventChain` 392ms to 238ms on 3000 events; `validateWorkGraph` 559ms to
  50ms on 5000 records against a 64-entry registry.
- Archive verbs no longer read up to 512 MiB of bundle bytes per invocation only to
  discard them.

## What this candidate deliberately does *not* do

**The engine's per-event collection scans are retained, not fixed.** The plan called
for indexing them. A measured baseline declined it: removing all four scans entirely
buys 4.0% of replay at the worst reachable point, and the shape that would make them
matter is unreachable — the 1 MiB canonical view-document limit binds at roughly
2000–3000 work records, long before the quadratic terms do. Four edits to the hottest
correctness path for 4% is a bad trade. What did land is instrumentation: the scans
are now counted, three new complexity fixtures pin each arm's closed form, and the
compaction-deferral proof — which claimed "there is no quadratic term" — is corrected
rather than left standing as a false statement.

**147 type errors remain.** A real `tsc` had never run against this repository. The
earlier figures in this note — 153 in one place and 167 in another, against 126
previously recorded — were estimates taken before a compiler was pinned, and they
disagreed with each other. The measured count is 147, across 14 files, from
`tsc --noEmit` on the repository `tsconfig.json` with TypeScript 5.9.3 and
`@types/node` 24.13.2 both pinned exactly. TypeScript 7.0.2 reports the same 147 at
the same files and lines. Three errors that made `workspace.ts` and the CLI
uncompilable are fixed; the rest are known debt, tracked, and not release-blocking —
the shipped build strips types and the runtime is unaffected. The compiler is now
pinned; wiring the typecheck gate into `verify:p1` waits on the count reaching zero.

**The per-mutation replay cost is unchanged and is the real scaling limit.**
`appendEvent` performs one full replay per mutation by design, so mutation latency is
280–416ms at 2500–4000 events and workspace construction is O(n²) overall. That is
roughly 40x larger than anything the collection scans contribute. It is documented,
not fixed.

## Verification

`pnpm verify:p1` runs the same 20 gates and passes. The claim ledger still binds 65
claims (13 framework-hygiene, 13 inertness-proof, 39 runtime-capability); no claim
was added, retired, or weakened to accommodate a fix.

Three fixes were confirmed against the built artifact before and after, rather than
by reading the patch:

| Defect | Baseline behaviour | After |
| --- | --- | --- |
| Knowledge claim leak | Claim left on disk; next mutation `KNOWLEDGE_LOCKED` | No claim; store usable |
| Orphaned recovery claim | `acquire` fails `WORKSPACE_LOCKED` permanently | Lease acquired, claim cleared |
| Enum coercion | `["specified"]`, `toString` object, boxed `String` all admitted | All rejected; legal values still admitted |

## Compatibility

No protocol, schema, or on-disk format change. Workspaces created by rc.5 are read
and replayed unchanged.

Two admissions that previously succeeded now fail, which is the direction that can
break a caller: enum fields no longer accept coercing values, and
`extension-registration.appliesTo` no longer accepts duplicate non-string entries.
Both were accepting malformed input. `docs/architecture/agent-integration-v1.md`
records all ten observable deltas for automated callers.

The companion `tcrn-workflow-helper` pins this repository by tag, commit, tree, and
tag object, so it must be re-pinned to this candidate before it will accept these
bytes.
